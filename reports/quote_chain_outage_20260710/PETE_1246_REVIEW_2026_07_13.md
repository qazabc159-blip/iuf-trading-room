## 4. Findings ŌĆö Priority Ranked

### ­¤ö┤ Blockers (must fix before ready)

1. **Label-honesty guarantee is bypassable through the pre-existing Admin manual-quotes endpoint ŌĆö the PR's own regression lock never exercises this path.**
   - Location: `apps/api/src/market-data.ts` ŌĆö `upsertManualQuotes()` (the one function in the `upsertKgiQuotes`/`upsertPaperQuotes`/`upsertTwseMisQuotes`/`upsertManualQuotes` family that does **not** pass a `sourceOverride` into `upsertProviderQuotes()`); reachable via `POST /api/v1/market-data/manual-quotes` (`server.ts`, Admin-gated). Root schema: `manualQuoteUpsertItemSchema.source = quoteSourceSchema.default("manual")`.
   - Failure scenario: `upsertProviderQuotes()` computes `const source = input.sourceOverride ?? item.source;`. Because `upsertManualQuotes()` never supplies `sourceOverride`, whatever the *caller* puts in `quotes[i].source` is written verbatim. Before this PR, `quoteSourceSchema` had 4 members and none of the "spoofable" ones mattered for the paper-mode gate exception (there wasn't one). After this PR, `"twse_mis"` is a 5th valid enum member that the *same* Zod schema now happily accepts on that same field. Any Admin-role caller (or a bug/script running with an Admin session) can `POST /api/v1/market-data/manual-quotes` with `{"quotes":[{"symbol":"2330","market":"TWSE","source":"twse_mis","last":9999,"timestamp":"<now>", ...}]}` and the resulting DB/cache row is indistinguishable from a real MIS-cron-written row ŌĆö it will trip `isPaperTrustedNonKgiSource` and auto-`allow` a paper order on a completely fabricated price, with a fabricated "official government feed" provenance label.
   - This is not a hypothetical: the PR's own report (`PAPER_QUOTEGATE_MIS_SOURCE_FIX_20260713.md`) explicitly names `POST /api/v1/market-data/manual-quotes` as the origin of "genuinely hand-typed Admin values" it is trying to distinguish from the MIS feed ŌĆö but never re-checks whether that same endpoint can now claim to *be* the MIS feed. Its "still blocked" regression-lock claim (T02/T05) only tests the *default* `source:"manual"` path, never a caller that explicitly sets `source:"twse_mis"` through `upsertManualQuotes`/the admin endpoint. That specific case is untested, and it defeats the stated safety property in exactly the words the PR uses to describe why the fix is safe ("a genuinely synthetic/hand-typed manual quote is unchanged ŌĆö still 'review', still blocked" ŌĆö true only if the caller doesn't set the one new field that unlocks the bypass).
   - Blast radius is scoped to the paper channel only (confirmed clean separation from execution/real-money ŌĆö see ┬¦A above) and requires an Admin-role session, so it is not a kill-switch/real-order violation. But it does defeat this PR's own stated regression lock and reopens exactly the "fake data labeled as trustworthy" gap the fix exists to close, which is a direct hit on the repo's own hard rule "õĖŹ fake µĢĖÕŁŚ" applied to data provenance, not just price display.
   - Suggested fix: make `upsertManualQuotes()` force `sourceOverride: "manual"` (mirroring every sibling function in the same file) so the admin endpoint can never write anything but a literal `"manual"` tag regardless of what the request body claims. Then add a T06-style test: `POST`-equivalent call to `upsertManualQuotes()` with an item whose `source` field is explicitly `"twse_mis"` must still land in the DB tagged `"manual"`.

### ­¤¤Ī Suggestions (should fix)

1. **Broader pre-existing issue surfaced during this review, out of this PR's diff ŌĆö escalate separately, does not block #1246.** The same unguarded `source` field on `manualQuoteUpsertItemSchema` already let an Admin-role caller spoof `source:"kgi"` (and `source:"tradingview"`) through `POST /api/v1/market-data/manual-quotes` *before* this PR ŌĆö `"kgi"` has been a valid enum member since before this diff, and `upsertManualQuotes()` has never forced an override.
