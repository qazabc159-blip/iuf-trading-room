## 卡點 2 之下的真正地雷：0055 繼承 0054 的同一個前提缺陷

`gh pr diff` 顯示 #1242 分支的 git 歷史裡，其實還帶著 #1241 原始的 0054 commit
（`924b4ac4`，PR body 自己也承認「疊在 Draft PR #1241 上」）。這代表：

1. **依賴關係**：0055 的 provenance-first backfill 第一段直接讀
   `iuf_events.workspace_id`（0054 加的欄位）。main 目前因 #1241 已被 revert，**不存在**
   這個欄位——若照原樣把 #1242 rebase 到目前 main 並 merge，這段 SQL 會直接
   `column "workspace_id" does not exist` 炸掉，crash-loop，複製一次 0054 的事故。這正是
   Mike 0054/0055/0056 審計報告自己點名的 Suggestion #3（「Merge-order dependency」）—
   Mike 當時的前提是「#1241 一定先 merge」，但 #1241 後來被 revert，前提已經不成立。
2. **更深層的地雷**：即使 0054 用未修正版本重新上（把已 revert 的內容原樣 cherry-pick
   回來），0055 自己的「孤兒 decision」fallback 邏輯，用的是跟 0054 一模一樣的
   `workspace_count <> 1 → RAISE` 判斷。Prod 現在確定有 18 個 workspace（`FIX_FORWARD_0054_SPEC`
   當天 railway ssh 查證的既有事實，本輪未重複查證——見下方「未直接查證項目」）。只要
   `iuf_decisions` 表裡有**任何一筆**孤兒列（trigger_type='event' 因為缺 0054 沒法解析、
   或 trigger_id 對應的 signal/event 列已被刪除），這段判斷必定 RAISE，複製同一種停機。
   OpenAlice M1 decision orchestrator（`iuf_events+signals → LLM → iuf_decisions`）自
   2026-06-24 起在 prod 跑了近 3 週，出現孤兒列的機率不低。

## 修法：兩個 migration 都套用 FIX_FORWARD_0054_SPEC 的 option (ii)

把「用 workspace 數量猜」換成「用寫死的 Primary Desk workspace UUID
（`888fd3bd-4a48-4656-9e6a-ac19360cc0de`，全 UUID 取自既有 QA 證據
`packages/qa-playwright/reports/prod_ai_market_contract_auth_20260529_013245Z/contract-audit-auth.json`，
與 spec 文件記的字首 `888fd3bd` 一致）」，只有在**那個特定 workspace 列本身不存在**時才
fail-closed——不受其他（含 17 個 4 月測試殘渣）workspace 數量影響：

- `packages/db/migrations/0054_iuf_events_workspace.sql`：孤兒 backfill DO block 改用
  `default_workspace_id CONSTANT UUID := '888fd3bd-...'`，`IF NOT EXISTS (SELECT 1 FROM
  workspaces WHERE id = default_workspace_id) THEN RAISE`。
- `packages/db/migrations/0055_iuf_decisions_workspace.sql`：同一個修法，套在孤兒
  decision 的 fallback 區塊。provenance-first 的兩段 UPDATE（讀 `iuf_events`／`signals`）
  維持原樣不動——因為 0054 已經在同一條 migration 鏈裡排在 0055 之前，欄位一定先存在，
  不需要額外的防呆 column-exists 判斷（Simplicity First：不加用不到的彈性）。
- `tests/ci.test.ts`：兩個 migration 的既有稽核鎖測試（`migration 0054 scopes...`／
  `migration 0055 scopes...`）原本斷言 `workspace_count <> 1` 這個字串，已同步改斷言新的
  `default_workspace_id CONSTANT UUID := '888fd3bd-...'` 與 fail-closed 檢查存在。

## 本機驗證（真跑 Postgres 16，非只 source-regex）

用 Docker 起一個乾淨的臨時 Postgres 16 容器（`localhost:15432`），完整跑過：

1. **全鏈 migrate**：`pnpm run migrate`（`PERSISTENCE_MODE=database`）從 0001 跑到 0055，
   全部 apply 成功，`Database schema is up to date.`。
2. **Positive path（模擬 prod 現況：18+ 個 workspace）**：down 0055 → down 0054 → seed 1
   個 id=`888fd3bd-...` 的 workspace ＋ 17 個雜訊 workspace（共 18+）＋ 1 筆孤兒

   forward 0054 → forward 0055 → **兩筆孤兒列的 `workspace_id` 都正確被指派為

## 卡點 2：base drift / mergeable=CONFLICTING — 修復手法

`git rebase origin/main` 直接跑會在 5 個檔案炸開衝突（`workspace-scope.ts` 甚至是
modify/delete 衝突），因為 rebase 機制偵測到 0054 的原始 commit（`924b4ac4`，內容與
main 歷史裡已 revert 的 `d45c5e17` 相同）「已經在上游出現過」而自動跳過，留下 0055 的
commit 在少了 0054 上下文的情況下硬套，才會全面衝突。改用**三段式 cherry-pick 重建**：

1. 從乾淨的 `origin/main`（`45a969a5`）開新分支。
2. `git cherry-pick d45c5e17`（0054 原始 merge 內容）→ 乾淨 apply（僅 `server.ts`／
   `tests/ci.test.ts` 自動合併，無衝突）。
3. `git cherry-pick d59af30e`（0055 原始內容，即 #1242 branch 舊 tip）→ 乾淨 apply。
4. `git cherry-pick <本輪修法 commit>`（0054/0055 pattern 修正＋測試更新）→ 乾淨 apply。
5. force-with-lease push 前後用 `git show origin/<branch>:<path> | grep
   "default_workspace_id CONSTANT"` 驗證修法字串確實在遠端內容裡（疊層 rebase 教訓
   `stacked-rebase-content-verify` 落地）。

Push 後 `gh pr view --json mergeable,mergeStateStatus` 從 `CONFLICTING`/`DIRTY` 轉
`mergeable=MERGEABLE`／`mergeStateStatus=CLEAN`。

## 誠實揭露

1. **範圍超出原始派工**：原任務只要求修 #1242（0055）。但 rebase 機制會機械式地把已被
   revert 的 0054 commit 重新帶回分支（因為 #1242 本來就疊在 #1241 上），若不修 0054 就
   等於把已知會炸 prod 的舊版本原封不動再 merge 一次。因此本輪把 Elva 同日
   `FIX_FORWARD_0054_SPEC_2026_07_13.md` 規劃中「0054 fix-forward」的部分，一併在
   #1242 分支內做掉了——這不是我主動擴大範圍，是 rebase 這個動作本身無法迴避的後果。
   **未另開 0054 獨立 PR**，兩個 migration 的修正都在同一支 #1242 分支、同一顆 commit
   裡，方便 Elva 決定要不要拆開審。
2. **未直接查證 prod 孤兒列數量**：試過用 `railway run`／`railway connect` 連 prod
   Postgres，確認 pg 服務**沒有對外暴露的 `DATABASE_PUBLIC_URL`**（`railway connect pg`
   明確回報找不到），代表 prod DB 沒有公開 TCP proxy——這是正確的安全姿態，本輪沒有嘗試
   新增一個來繞過（會擴大 prod DB 的攻擊面，超出這個任務該做的事）。因此「prod 到底有
   幾筆孤兒 decision」沒有直接數字，修法的必要性建立在兩個已查證的事實上：(a) prod 確定
   有 18 個 workspace（`FIX_FORWARD_0054_SPEC` 同日 railway ssh 唯讀查證的既有紀錄，非
   本輪重新查證）；(b) 只要有 ≥1 筆孤兒列，原邏輯必炸——18≠1 這個事實本身就已經讓原本的
   `workspace_count <> 1` 判斷失效，不需要知道確切孤兒列數才能判斷這是真風險。
3. **`FIX_FORWARD_0054_SPEC_2026_07_13.md` 與 Mike 審計報告目前是主 repo 工作目錄裡的
   untracked 檔案**（`git status` 確認未 commit），我的 migration SQL 註解裡引用了這個
   路徑當人類可讀的脈絡指標——若該檔案之後被移動或改名，註解裡的路徑會過期，但這不影響
   migration 本身的邏輯正確性。
4. **未觸碰** Railway `EXPECTED_MIGRATION_COUNT`、未部署、未 merge——照 PR 原本「未解決
   事項」與本任務指示，這些留給 Elva／實際執行 merge 的人處理。
