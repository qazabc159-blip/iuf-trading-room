## 問題 A — F-AUTO positions shares=0 + 2/8 priced

### 根因（程式層）

**Bug 1 (Critical): `??` 運算子對 `0` 無效**

- 檔案：`apps/api/src/s1-sim-runner.ts` 第 1054 行（fix 後：第 1061 行）
- 舊碼：`shares: r.filled_shares ?? r.shares`
- 根因：KGI SIM 從不回傳 broker fill 確認，`reconcileKgiOrder()` 回傳 `filledQty=0`（數字零，非 `null`）。JavaScript `??` 只替換 `null / undefined`，不替換 `0`。
  - `0 ?? 18000` → `0`（返回 0，bug）
  - `null ?? 18000` → `18000`（正確）
- 因此 8 個訂單的 `filled_shares=0`，套公式後 `shares=0` 全滅。
- 市值計算：`market_value_twd = shares * last_price = 0 * close = 0` → 全部 0。

**prod 證據（2026-06-24T04:07Z）：**
```json
GET /api/v1/portfolio/f-auto
→ positions[0] { symbol:"5468", shares:0, avg_cost:33, last_price:36.3,
                 unrealized_pnl_twd:0, market_value_twd:0 }
   ...所有 8 檔均 shares=0
→ data_source: "audit_log_rebuild"
→ notes: ["s1_audit_primary: rebuilt 8 S1 holdings from 2026-06-23 submitted SIM orders"]
```

**basket 對照（正確的 target_shares）：**
```json
GET /api/v1/internal/s1-sim/basket?date=2026-06-23
→ 5468 target_shares=18000, 2492 target_shares=1000, 6654 target_shares=2000 ...
```

**Bug 2 (Transient): TPEX EOD 2/8 priced**

- Bruce 03:51 拿到 `notes["tpex_eod_unavailable"]` → 只 2/8 有價。
- 筆者 04:07 再查已是 `notes["mark_to_market: last_price for 8/8 positions"]` → 8/8。
- 根因：TPEX mainboard close 快取（`getTpexMainboardCloseRows` 10s timeout, 5min TTL）在 03:51 快取剛好過期且重取失敗（TPEX OpenAPI 深夜有機率 timeout），04:07 已重取成功。
- 這是瞬態快取問題，不是常態 bug。TPEX 10s timeout + 5min TTL 是 PR #1065 已修好的版本。無需額外修碼。

### 修法

**`apps/api/src/s1-sim-runner.ts` — 兩處改動：**

1. **shares 解析（第 1061 行）**：
   ```typescript
   // 舊碼（bug）
   shares: r.filled_shares ?? r.shares,
   
   // 新碼（fix）
   shares: (r.filled_shares != null && r.filled_shares > 0) ? r.filled_shares : r.shares,
   ```
   當 `filled_shares=0`（SIM unconfirmed）→ 使用 basket `target_shares`（正確的委託張數）。
   當 `filled_shares>0`（真實成交）→ 優先使用成交量。
   當 `filled_shares=null`（舊資料）→ fallback 到 `target_shares`。

2. **`orders_accepted` 計數（第 953 行）**：
   同次 PR 一起修（見問題 B 說明）。

**回歸風險：極低。**
- `buildS1PositionsSnapshot()` 是 read-only SIM path，不碰下單。
- 只改 `?? ` 的 falsy 邊界條件，不改任何 DB/file 讀取路徑。
- 如果未來有真實 fill（`filled_shares>0`），fix 碼仍正確優先使用真實成交量。

## 問題 B — S1 6/23 orders_accepted=0

### 根因（程式層）

**Cosmetic counter bug — 訂單實際都送進去了。**

- 檔案：`apps/api/src/s1-sim-runner.ts` 第 950 行（fix 後：第 953 行）
- 舊碼計數集合：`["accepted", "filled", "partially_filled", "cancelled"]`
- KGI SIM 行為：所有 SIM 訂單 `createOrder()` 成功（`accepted=true`），但後續 `reconcileKgiOrder()` poll 3 次沒拿到 fill/deal report → `brokerReportConfirmed=false` → `brokerStatus="unconfirmed"`。
- 最終 `results[]` 的 `status` 全是 `"unconfirmed"`，不在計數集合內 → `orders_accepted=0`。

**但持倉重建是對的：**
- `buildS1PositionsSnapshot()` 的 `HELD_STATUSES` 包含 `"unconfirmed"` → 8 個 unconfirmed 訂單都被算作 held → 8 個 positions 被重建（證明訂單確實到了 KGI SIM）。

**prod 證據（2026-06-24T04:07Z）：**
```json
GET /api/v1/internal/s1-sim/status
→ today_orders: {
    date: "2026-06-23",
    submitted_at_tst: "2026-06-23T09:04:30+08:00",  // 在窗內 09:00-09:20
    orders_attempted: 8,
    orders_accepted: 0,   // 計數 bug
    orders_rejected: 0
  }
→ today_eod: { position_count: 8, data_source: "audit_log_rebuild" }
  // 8 個 positions 被重建 → 訂單確實被 KGI SIM 接受
```

**結論：這是 cosmetic 計數 bug（B 類低風險），不是真故障。**

### 修法

**`apps/api/src/s1-sim-runner.ts` 第 953 行：**
```typescript
// 舊碼（bug）
orders_accepted: orderResults.filter((r) =>
  ["accepted", "filled", "partially_filled", "cancelled"].includes(r.status)
).length,

// 新碼（fix）
// KGI SIM never returns a broker fill report — submitted orders stay "unconfirmed"
// forever. Count "unconfirmed" as accepted (submission reached the broker) so the
// counter reflects real activity rather than showing 0 when 8 orders went through.
orders_accepted: orderResults.filter((r) =>
  ["accepted", "filled", "partially_filled", "cancelled", "unconfirmed"].includes(r.status)
).length,
```

**回歸風險：極低。**
- 只改計數器，不改 `HELD_STATUSES` 或任何下單路徑。
- `rejected` / `skipped` 依然被排除（測試 `S1-ACCEPTED-FIX-1` 驗證）。
