## 🔴 Blocker 1 — double-submission race in `runV51OrderSubmitTick()`

**Problem** (Pete's finding): `hasAlreadySubmitted()` is an async DB `SELECT` against
`audit_logs` and was the *only* protection. `setInterval` (via the scheduler wrapper) does not
wait for the previous async callback to resolve, and the 08:20–08:40 TST window structurally
allows two 15-min poll ticks to land inside it. If a second tick's `hasAlreadySubmitted()` check
ran before the first tick's `writeAuditRecord()` had committed, both ticks would pass and submit
the full 30-name / 10M-notional basket twice on the very first live day.

**Fix**: Added a synchronous, awaits-free in-memory "claim today's slot" guard at the very top of
`runV51OrderSubmitTick()`, set before any `await` — mirroring `s1-sim-runner.ts`'s
`_orderSubmitLastFiredDate` pattern (the exact precedent this module already reuses order-submit
mechanics from).

- `apps/api/src/v51-sim-basket-runner.ts`:
  - New module-level state `_v51OrderSubmitLastFiredDate` (mirrors S1's naming/pattern).
  - New exported `_v51ClaimOrderSubmitTickForDate(todayTst): boolean` — synchronous claim,
    returns `false` if the date was already claimed (closes the race).
  - New exported `_v51ReleaseOrderSubmitGuard(): void` — releases the guard.
  - `runV51OrderSubmitTick()` now calls the claim function as its very first line (before the
    `listEmbeddedV51BasketSignalDates()` await). The pre-existing `audit_logs` check remains in
    place unchanged — it is still the only guard that survives a redeploy (in-memory guard resets
    to `""` on process restart).
  - Guard is **released** (allowing a later tick in the same 08:20–08:40 window to retry) on two
    known-transient failure signatures: `credentials_missing` and `login_failed`. This matters in
    practice — the EC2 KGI gateway opens on the *same* 08:20 EventBridge schedule as this
    pipeline's window starts, so the very first tick could plausibly hit a not-yet-reachable
    gateway; without the reset, that would silently kill the entire day's entry. Guard is **not**
    released on `fail_closed_basket_read` / `fail_closed_subscription_cap` (permanent for the day
    — the basket file won't appear mid-day, and the cap check result won't change).
  - Updated the function's JSDoc to describe the two-layer guard accurately (the prior JSDoc
    over-claimed "idempotent via audit-log check" alone, which Pete's review pointed out was
    weaker than the S1 precedent it referenced).

**Tests added** (`apps/api/src/v51-sim-basket-runner.test.ts`, +3, real behavioral tests against
the exact claim/release primitive the runner calls, not source-regex):
1. `order-submit guard: overlapping tick for the same entry date — only the first claim succeeds`
   — simulates two overlapping ticks; asserts first claim `true`, second `false`.
2. `order-submit guard: a different entry date is not blocked by a prior day's claim` — confirms
   the guard doesn't accidentally block a legitimate future basket (Pete's re-review criterion b).
3. `order-submit guard: release allows a subsequent tick to retry the same date` — confirms the
   retry-reset path mirrors S1's convention (Pete's re-review criterion a: "resets appropriately").

## Addendum — Elva's merge-review residual gap (2026-07-13, same-day follow-up commit)

Elva caught one more real gap while verifying blocker 1's fix before merge: in
`submitV51BasketOrders()`, `await writeJson(...)` (the human-readable report JSON) ran *after* the
order-submit loop but *before* `writeAuditRecord()`, with no error handling. If `writeJson` threw
(container fs issue — rare but real) after orders had already been submitted to KGI SIM, the
exception would propagate up to `runV51OrderSubmitTick()`'s `catch`, which unconditionally calls
`_v51ReleaseOrderSubmitGuard()` — with no `audit_logs` row ever written. The next tick's
`hasAlreadySubmitted()` would then return `false`, and the whole already-filled 30-name basket
would be re-submitted. Elva confirmed the per-order submit loop itself cannot throw (verified
segment by segment), so this was the one remaining seam.

**Fix (Elva's option A — minimal, recommended)**: wrapped the report-JSON write in a new exported
`_v51WriteReportJsonBestEffort()` helper — failure is caught, recorded into `failsafeNotes` (same
array reference already embedded in the report passed to `writeAuditRecord()`), and never
propagates. `writeAuditRecord()` already swallows its own DB errors internally (pre-existing), so
`submitV51BasketOrders()` can no longer throw once order submission has started — an audit row is
now always attempted. Added a comment on the outer `catch` in `runV51OrderSubmitTick()` explaining
its unconditional guard release is now only reachable by a *pre-submission* throw (no orders
placed yet), where retrying the whole basket from scratch on the next tick is safe.

**Tests** (+2, against the real `writeJson()` failure path, not a mock — a NUL-byte path is a
genuine, portable fs failure on every platform): (1) failure is caught, recorded, does not throw;
(2) success path writes a real file via a real temp dir and leaves `failsafeNotes` untouched.
