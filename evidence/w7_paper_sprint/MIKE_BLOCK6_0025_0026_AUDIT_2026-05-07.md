## MIGRATION 0025 — iuf_events

### 1. Migration Intent
- Goal: Create iuf_events table for OpenAlice event rule engine (BLOCK #6)
- Stores triggered rule events: severity / ticker / payload / ack status
- Dedup guard in code queries this table (rule+ticker within 1h window)
- Sprint task: BLOCK #6 B event rule engine (PR #267)
- forward + down pair: PRESENT (both in branch)

### 2. Schema Changes Summary
New tables:
- iuf_events (id UUID PK, rule_id TEXT NOT NULL, rule_name TEXT NOT NULL,
  severity TEXT NOT NULL CHECK IN ('info','warning','critical'), ticker TEXT NULL,
  payload JSONB NOT NULL DEFAULT '{}', triggered_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  acknowledged BOOLEAN NOT NULL DEFAULT FALSE)
- _quarantine_iuf_events (id UUID PK, original_id UUID NOT NULL, rule_id TEXT NOT NULL,
  reason_code TEXT NOT NULL, quarantined_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  payload JSONB NOT NULL DEFAULT '{}')

New indexes:
- iuf_events_rule_ticker_time_idx ON (rule_id, ticker, triggered_at DESC)
- iuf_events_triggered_at_idx ON (triggered_at DESC)
- iuf_events_unread_idx ON (acknowledged, triggered_at DESC) WHERE acknowledged = FALSE (partial)

New FKs: NONE
Drops: NONE

## MIGRATION 0026 — iuf_notification_preferences

### 1. Migration Intent
- Goal: Per-user notification preference table for OpenAlice alert system
- Replaces global DIGEST_EMAIL env-var with per-user preference row
- FK to users(id) ON DELETE CASCADE
- Sprint task: BLOCK #6 C email digest per-user prefs (PR #268)
- forward + down pair: PRESENT

### 2. Schema Changes Summary
New tables:
- iuf_notification_preferences (id UUID PK, user_id UUID NOT NULL FK→users(id) CASCADE,
  email_enabled BOOLEAN NOT NULL DEFAULT TRUE, email_address TEXT NOT NULL,
  min_severity TEXT NOT NULL DEFAULT 'warning' CHECK IN ('info','warning','critical'),
  digest_schedule TEXT NOT NULL DEFAULT 'daily_close' CHECK IN ('daily_close','realtime'),
  rule_filter JSONB NULL, created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), UNIQUE(user_id))
- _quarantine_iuf_notification_preferences (id UUID PK, original_id UUID NOT NULL,
  user_id UUID NOT NULL, reason_code TEXT NOT NULL, quarantined_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  payload JSONB NOT NULL DEFAULT '{}')

New indexes:
- iuf_notif_prefs_user_idx ON (user_id)

New FKs:
- user_id → users(id) ON DELETE CASCADE

Drops: NONE

### 3. IUF Audit Checklist (A-G)

#### A. Forward + Down Pair
- [PASS] 0026_iuf_notification_preferences.down.sql present on disk
- [PASS] down.sql drops _quarantine table first, then iuf_notification_preferences (reverse dep order)
- [PASS] both DROPs use IF EXISTS
- [PASS] forward → down → forward is clean (CREATE IF NOT EXISTS / DROP IF EXISTS)

#### B. Foreign Key Cascade
- [PASS] FK on user_id → users(id) explicitly states ON DELETE CASCADE
- [PASS] parent table users exists (0001_initial.sql — established in earliest migration)
- [REVIEW — SEE FINDING #1] CASCADE semantics: if a user is deleted, their notification
  preference row is auto-deleted. This is correct behavior for a preference table (no orphan prefs).
  No multi-level cascade risk (iuf_notification_preferences is a leaf table with no children).
- [PASS] No implicit default cascade — explicit declaration present

#### C. Idempotency / UNIQUE Constraint
- [PASS] UNIQUE (user_id) enforced at DB level — one preference row per user
- [FINDING #2] iuf_notif_prefs_user_idx on (user_id) is REDUNDANT with the UNIQUE(user_id) constraint.
  In PostgreSQL, a UNIQUE constraint automatically creates a B-tree index on the constrained column(s).
  The separate CREATE INDEX on user_id adds write overhead with zero query benefit.
  This is anti-pattern P6 (redundant UNIQUE + regular index on same columns).
  Flag: SUGGESTION (not blocker) — consistent with how we handled 0024.

#### D. Index Coverage
- [PASS] Primary query path: lookup by user_id — covered by UNIQUE constraint index
- [REDUNDANT — see Finding #2] iuf_notif_prefs_user_idx is superfluous
- [PASS] Table is expected to have O(1) rows per user (single user system currently) —
  no additional composite indexes needed

#### E. NOT NULL / DEFAULT / Data Backfill
- [PASS] New table — no existing rows, no backfill needed
- [FINDING #3 — BLOCKER] email_address TEXT NOT NULL has NO DEFAULT.
  The table uses CREATE TABLE IF NOT EXISTS but this is a fresh table (no rows yet).
  However, if a future migration or worker tries to INSERT without supplying email_address,
  the NOT NULL constraint with no DEFAULT will cause a runtime error.
  MORE CRITICALLY: the current code (openalice-email-digest.ts) does NOT read from this table.
  The digest uses DIGEST_EMAIL env var directly. The table is future infrastructure with no
  active writer. This means:
  - If a user row is created (e.g., via invite flow) and a worker tries to auto-create a
    preference row without email_address, it will fail with a NOT NULL violation.
  - There is no documented backfill or seed plan for existing users.
  Recommendation: Jason should either:
    A) Add DEFAULT '' and validate at application layer, OR
    B) Document explicitly that email_address is caller-supplied and no auto-insert worker exists yet.
  At current state (table written but not read by any active worker), this is LOW operational risk
  but a design clarity gap. Classifying as SUGGESTION rather than BLOCKER since no active writer exists.

- [PASS] updated_at has DEFAULT NOW() — but NOTE: no trigger or application-layer logic to
  auto-update updated_at on row updates. If the table is used via raw UPDATE statements,
  updated_at will stay stale. Acceptable at DRAFT stage — flag for Jason to add trigger or
  app-layer update when worker is implemented.

- [PASS] All other NOT NULL columns have correct DEFAULTs:
  - email_enabled DEFAULT TRUE
  - min_severity DEFAULT 'warning'
  - digest_schedule DEFAULT 'daily_close'
  - created_at DEFAULT NOW()
  - updated_at DEFAULT NOW()

#### F. IUF Hard Rules
- [PASS] No DROP DATABASE / TRUNCATE on production tables
- [PASS] No environment-specific logic
- [PASS] No data manipulation on existing rows
- [PASS] No secrets / tokens / credentials in SQL or comments
- [PASS] 0026 is consecutive after 0025 (correct sequence)
- [PASS] DIGEST_EMAIL default in code (qazabc159@gmail.com) is NOT written into the SQL migration

#### G. Migration File Hygiene
- [PASS] Numbering: 0026 is consecutive after 0025
- [PASS] File name clearly describes intent: 0026_iuf_notification_preferences
- [PASS] Comment header has purpose, owner, sprint reference, related file
- [NOTE] No "-- W7 Day..." sprint marker format (same nit as 0025)
- [PASS] DRAFT infix present, migrate.ts filter blocks auto-apply
- [PASS] down.sql is present and paired

### 4. Findings — Priority Ranked

#### BLOCKERS
NONE

#### Suggestions
1. [CASCADE DELETE]: ON DELETE CASCADE on user_id is correct for a preference table but
   should be explicitly documented in code that calls user deletion, so operators know
   the cascade exists. Not a schema blocker — just a documentation gap.

2. [email_address NOT NULL NO DEFAULT]: The email_address column has no DEFAULT and no
   active auto-insert writer. If a worker is added in a future sprint that auto-creates
   preference rows on first login, it must explicitly supply email_address or the INSERT
   will fail. Jason should document the insertion contract before adding any auto-insert
   logic. Current state: table exists, no active writer → operationally safe now.

3. [updated_at no auto-update trigger]: updated_at defaults to creation time and will
   not auto-update on subsequent row modifications without application-layer logic or a
   DB trigger. If this table becomes actively mutated (user changes prefs), add a trigger
   or explicit UPDATE updated_at = NOW() in the update path.

#### Nits
1. [P6 REDUNDANT INDEX]: iuf_notif_prefs_user_idx ON (user_id) is redundant with UNIQUE(user_id).
   PostgreSQL already creates an implicit B-tree index for the UNIQUE constraint.
   Recommend Jason remove the explicit CREATE INDEX at promote time.
   Evidence: Anti-pattern P6 in Mike's playbook (same as 0024 review finding).

2. Sprint marker nit: same as 0025 — no "-- W7 Day..." format.

### 5. Rollback Dry-Run Plan

5. Estimated rollback time: < 5 seconds
Data loss risk: any preference rows stored are lost on rollback. Currently no rows
exist and no active writer, so data loss risk = zero at current state.
