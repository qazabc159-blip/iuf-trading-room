## Systemic issue (≥3 PR pattern)

**Status-color tone rename chain (#146 → #147 → #166)**: three PRs over the 5/4-5/5 window where one rename (`up`/`down` → `status-ok`/`status-bad` for governance status) was rolled out without:
- a single all-repo grep evidence block,
- a regression test asserting tone propagation across all consumers,
- a single coherent commit instead of three serialized cleanups.

This is the exact failure mode governance Rule 2 was written to prevent (see `feedback_governance_rules_2026-04-22.md`). Codex's pattern is **rapid 25-min auto-loop pass-by-pass refinement**, which structurally trades single-shot completeness for high cadence. That trade is acceptable for purely-visual polish but **unacceptable for any string-keyed semantic rename**.

Recommendation to Codex (filed; not posted to PR threads):

1. Before any string-keyed rename across `apps/web/**` (tone tokens, status enums, source-state strings, blockedReason), run `rg '"<old>"'` across `apps/web` AND `apps/api` AND `packages/`, paste hit list into PR body, fix all in one PR.
2. Add at least one regression test or type-narrowing change that **mechanically forbids** the old token from compiling. E.g., for `Tone` the type was widened in #146 but not narrowed to forbid mixing `up` (market) with `status-ok` (governance) at the same call site. A discriminated `MarketTone | StatusTone` would make the next miss a typecheck failure.
3. For evidence-only PRs, sanity-check screenshot blob uniqueness. A 1-line script `if (Get-FileHash a) -eq (Get-FileHash b)` in the smoke harness would have caught #177.
